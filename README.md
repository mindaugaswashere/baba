# 🔐 Full-Stack Auth Starter

User **registration + login** with a React/TypeScript front end, a Rust (Axum)
API, and PostgreSQL. Sessions are carried in a **secure, httpOnly cookie** (no
tokens in `localStorage`), passwords are hashed with **Argon2id**, and the whole
thing runs locally with **one command per side**.

> The original tiny `baba` (a hand-written HTTP server) still lives in
> [`src/`](src/) + the root `Cargo.toml` and is unrelated to this app. The auth
> app lives entirely in [`backend/`](backend/) and [`frontend/`](frontend/).

---

## The flow

```
Guest landing (/)  ──►  Register / Login  ──►  Logged-in dashboard (/)
       ▲                                                  │
       └──────────────────────  Log out  ◄───────────────┘
```

- `/` is the **same path for everyone**: guests see a landing page, authenticated
  users see their dashboard (with their account "user table").
- `/login` and `/register` redirect to `/` if you're already signed in.
- Auth state is established on load by calling `GET /api/auth/me` with the cookie.

## Architecture

| Layer | Tech | Notes |
|-------|------|-------|
| Frontend | React 18 + TypeScript + Vite | React Router, react-hook-form + zod validation. Dev server on **:5174**, proxies `/api` → backend. |
| Backend | Rust + **Axum 0.8** | `/api/auth/*` JSON API. Listens on **:8080**. |
| Auth | **Argon2id** + **JWT (HS256)** in an httpOnly/SameSite cookie | Constant-time login to resist user enumeration. |
| Database | **PostgreSQL 16** via **SQLx 0.8** | A *project-local* cluster (own data dir + port **5433**), no Docker, no sudo. |

Because the browser only ever talks to the Vite dev server (:5174) and `/api` is
**proxied** to the backend, the auth cookie stays same-origin — no CORS dance.

## Quick start

Three terminals (or run the first two once, they persist):

```bash
# 1) Database — creates a project-local cluster + role + db + backend/.env (idempotent)
cd backend
./scripts/db.sh init

# 2) Backend API  (http://127.0.0.1:8080)
cargo run            # still in backend/

# 3) Frontend      (http://127.0.0.1:5174)
cd ../frontend
npm install          # first time only
npm run dev
```

Then open **http://127.0.0.1:5174** → Sign up → you land on the dashboard.

Stop the database cluster when you're done: `cd backend && ./scripts/db.sh stop`.

## API reference

Base path: `/api`. All auth endpoints set/clear the `auth_token` cookie.

| Method | Path | Body | Success | Errors |
|--------|------|------|---------|--------|
| `POST` | `/auth/register` | `{ email, password }` | `201 { user }` + cookie | `422` invalid, `409` email taken |
| `POST` | `/auth/login` | `{ email, password }` | `200 { user }` + cookie | `401` bad credentials |
| `POST` | `/auth/logout` | — | `204` + clears cookie | — |
| `GET`  | `/auth/me` | — (cookie) | `200 { user }` | `401` not authenticated |
| `GET`  | `/health` | — | `200 "ok"` | — |

`user` = `{ id, email, createdAt }`. Emails are trimmed + lowercased; passwords
must be 8–128 characters.

Try it directly:

```bash
curl -i -c jar -X POST http://127.0.0.1:8080/api/auth/register \
  -H 'Content-Type: application/json' \
  -d '{"email":"you@example.com","password":"supersecret"}'
curl -b jar http://127.0.0.1:8080/api/auth/me
```

## Configuration

Everything the backend needs is in `backend/.env` (auto-generated, gitignored).
See [`backend/.env.example`](backend/.env.example) for the template.

| Var | Purpose |
|-----|---------|
| `DATABASE_URL` | Postgres connection (local cluster, :5433). |
| `JWT_SECRET` | HS256 signing key (generated, 64 hex chars). |
| `JWT_TTL_SECONDS` | Session lifetime (default 7 days). |
| `BIND_ADDR` | API listen address (`127.0.0.1:8080`). |
| `COOKIE_SECURE` | `true` to mark the cookie HTTPS-only (set in prod). |
| `CORS_ORIGIN` | Allowed cross-origin caller (the Vite dev server). |

## Security notes & production checklist

This was hardened with an adversarial security review (17 confirmed findings;
the worthwhile ones are fixed, the rest are listed below for production).

**Already done:**

- Argon2id password hashing (OWASP-default params), run off the async runtime.
- Session JWT in an **httpOnly + SameSite=Lax** cookie — not `localStorage`,
  so XSS can't read it and cross-site POSTs don't carry it (CSRF mitigation).
- JWT pinned to **HS256** (no `alg=none`/confusion), with enforced `exp`.
- **Parameterized SQL** everywhere (no string-built queries).
- **Constant-time login** (decoy hash for unknown emails) to blunt enumeration.
- **Generic error responses** — internal details are logged, never returned.
- Secrets (`JWT_SECRET`, `DATABASE_URL`) are **redacted from `Debug`/logs**, and
  a too-short `JWT_SECRET` is rejected at startup.
- **64 KiB request-body cap** and a **DB acquire timeout** to resist trivial DoS.

**Before going to production:**

- [ ] Serve over **HTTPS** and set `COOKIE_SECURE=true`.
- [ ] Add **rate limiting / lockout** on `/auth/login` and `/auth/register`
      (skipped here on purpose: behind the dev proxy every request looks like it
      comes from `127.0.0.1`, so per-IP limiting would throttle everyone).
- [ ] Add **CSRF tokens** as defense-in-depth (SameSite=Lax covers the common
      case but not everything, e.g. if you ever relax it to `None`).
- [ ] Decide on **session revocation** — JWTs are stateless, so logout only drops
      the cookie; a stolen token is valid until `exp`. Add a token version or a
      short-lived access + refresh-token scheme if you need server-side revocation.
- [ ] Add **security headers** (HSTS, CSP, X-Frame-Options, X-Content-Type-Options).
- [ ] The **register** endpoint returns `409` for a taken email (good UX, but it
      does reveal that an address exists). Switch to a generic flow if enumeration
      resistance matters more than UX for you.
- [ ] Move the DB to a managed/standalone Postgres and rotate the generated secrets.

## Troubleshooting

- **Backend won't start / DB errors** → `cd backend && ./scripts/db.sh status`; if
  stopped, `./scripts/db.sh start`. Inspect data with `./scripts/db.sh psql`.
- **Port already in use** → backend `BIND_ADDR` (`.env`) and frontend port
  (`vite.config.ts`) are both configurable; the DB port is `5433` in `db.sh`.
- **Login always fails after editing `.env`** → the `JWT_SECRET` changed, which
  invalidates existing cookies; just log in again.
